📋 Guide de Soutenance — Système de Gestion des Clubs IHEC

🌍 Description Non-Technique (Explication Simple)

C'est quoi cette application ?

C'est un logiciel de gestion des clubs universitaires de l'IHEC Carthage. Il permet à 3 types d'utilisateurs de gérer les clubs, les membres et les événements :
Qui ?Quoi ?
|---|---|
**L'Administrateur**Crée des clubs, gère tous les membres, supervise les budgets
**Le Président**Gère les événements de son club, accepte/refuse les propositions, surveille le budget
**L'Étudiant**Consulte les événements et propose ses propres idées d'activités

Comment ça marche concrètement ?

1. L'utilisateur lance l'application → une fenêtre de connexion apparaît 2. Il entre son email et mot de passe 3. Selon son rôle, il accède à un espace différent (Admin, Président, ou Étudiant) 4. Toutes les données sont sauvegardées en ligne dans une base de données PostgreSQL hébergée sur Supabase (cloud)

🔷 Architecture en Couches (Layers)

Le projet suit le pattern MVC enrichi avec 4 couches :

1. model/ — La couche Modèle (Entités JPA)

Ce sont les objets métier qui représentent les données.
  • Membre → classe mère abstraite (Admin, President, Etudiant)
  • Club → entité centrale
  • Budget → composition dans Club
  • Activite → classe abstraite pour les événements
  • EvenementAcademique, EvenementSportif, EvenementCulturel → sous-classes
  • EvenementProposition → proposition en attente de validation

2. repository/ — La couche Accès aux Données

Ce sont les interfaces qui parlent directement à PostgreSQL.
  • MembreRepository extends JpaRepository → CRUD automatique
  • ClubRepository, ActiviteRepository, BudgetRepository
  • Spring génère automatiquement toutes les requêtes SQL

3. service/ — La couche Logique Métier

Contient les règles de gestion de l'application.
  • ClubService → créer club, ajouter/supprimer membres, etc.
  • MembreService → authentification, inscription
  • EvenementService → créer événement, accepter/refuser proposition, déduire du budget

4. controller/ — La couche API REST

Expose les endpoints HTTP utilisés par l'interface Swing.
  • ClubController → /api/clubs
  • MembreController → /api/auth/login, /api/auth/register
  • EvenementController → /api/clubs/{id}/evenements, /api/propositions
  • BudgetController → /api/clubs/{id}/budget

5. gui/ — L'Interface Swing

La fenêtre graphique qui appelle l'API via ApiClient.java.
  • MainGUI → fenêtre principale + écran de connexion
  • AdminPanel, PresidentPanel, EtudiantPanel
  • ApiClient → envoie les requêtes HTTP et parse le JSON
  • ExportUtils → export CSV des tableaux

🧬 Concepts POO Appliqués

1. 🔒 Encapsulation

Tous les champs des classes sont private, accessibles uniquement via des getters/setters. 

// Budget.java
private double solde;
public double getSolde() { return solde; }
public void ajouterDepense(double montant) { solde -= montant; }

2. 🧱 Héritage (Inheritance)

La hiérarchie des membres utilise SINGLE_TABLE avec un discriminateur : 

Membre (classe mère)
├── Admin        → role = "admin"
├── President    → role = "president"
└── Etudiant     → role = "etudiant"
La hiérarchie des événements : 

Activite (classe abstraite)
├── EvenementAcademique  → type = "academique"
├── EvenementSportif     → type = "sportif"
├── EvenementCulturel    → type = "culturel"
└── EvenementProposition → type = "proposition"

3. 🎭 Polymorphisme

La méthode abstraite getCoutTotal() est redéfinie différemment dans chaque sous-classe d'événement : 

// Activite.java (abstraite)
public abstract double getCoutTotal();

// EvenementAcademique → retourne coutGlobal
// EvenementSportif    → retourne coutEquipement + coutLogistique
// EvenementCulturel   → retourne coutDecoration + coutSonorisation
Grâce au polymorphisme, le service peut appeler evenement.getCoutTotal() sans connaître le type exact.

4. 🧩 Abstraction

  • Activite est une classe abstraite → on ne peut pas créer un objet Activite directement
  • Elle force les sous-classes à implémenter getCoutTotal()
  • Membre est aussi une classe abstraite en pratique (constructeur protected)

5. 🔗 Composition vs Association

RelationExempleType JPA
|---|---|---|
**Composition**Club → Budget`@OneToOne` cascade ALL (le budget n'existe pas sans club)
**Association**Club ↔ Membres`@ManyToMany` (un membre peut être dans plusieurs clubs)
**Association**Club → Événements`@OneToMany` (les événements appartiennent à un club)

🔄 Flux de Données — Exemple Complet

Scénario : Un étudiant propose un événement


1. EtudiantPanel.java
   → ApiClient.post("/clubs/1/propositions", { titre, date, lieu, explication })

2. EvenementController.java (@PostMapping)
   → Reçoit la requête HTTP
   → Crée un EvenementProposition avec statut = "EN_ATTENTE"

3. EvenementService.java
   → Sauvegarde via ActiviteRepository

4. PostgreSQL (Supabase)
   → INSERT INTO activite (type_evenement='proposition', statut='EN_ATTENTE', ...)

5. AdminPanel ou PresidentPanel clique "Gérer propositions"
   → GET /api/clubs/1/propositions

6. Président clique "Accepter"
   → PUT /api/propositions/5/accepter
   → EvenementService.accepterProposition()
   → Change statut à "APPROUVE"
   → Déduit le coût du budget du club

💡 Questions Probables du Professeur & Réponses

Q1 : Pourquoi avez-vous utilisé SINGLE_TABLE pour l'héritage ?

R : C'est la stratégie la plus performante car elle utilise une seule table pour toute la hiérarchie. Les requêtes n'ont pas besoin de jointures. Le discriminateur role (pour les membres) ou type_evenement (pour les activités) identifie la sous-classe.

Q2 : Qu'est-ce que le polymorphisme dans votre projet ?

R : La méthode getCoutTotal() est abstraite dans Activite. Chaque événement la calcule différemment : académique = coût global fixe, sportif = équipement + logistique, culturel = décoration + sonorisation. Le service appelle evenement.getCoutTotal() sans savoir quel type c'est.

Q3 : Quelle est la différence entre Composition et Association ?

R :
  • Composition (Club → Budget) : le budget n'existe pas sans le club. Si on supprime le club, le budget est supprimé automatiquement (cascade = ALL, orphanRemoval = true).
  • Association (Club ↔ Membre) : un membre peut exister sans club et appartenir à plusieurs clubs. C'est un @ManyToMany.

Q4 : Comment fonctionne l'authentification ?

R : On envoie POST /api/auth/login avec email et password en JSON. Le service cherche le membre par email dans la base, appelle membre.authentifier(email, mdp) qui compare les champs. Si correct, on retourne les infos du membre avec son rôle. Le client Swing stocke l'objet utilisateur courant en mémoire.

Q5 : Pourquoi Spring Boot et non une appli Swing pure ?

R : Spring Boot offre un serveur REST qui permet de persister les données dans une base cloud (Supabase). L'interface Swing est ainsi déconnectée du stockage — toute la logique métier est côté serveur. C'est une architecture client-serveur moderne.

Q6 : C'est quoi un Repository ?

R : Un Repository (ex : ClubRepository extends JpaRepository) est une interface Spring Data qui génère automatiquement toutes les requêtes SQL (SELECT, INSERT, UPDATE, DELETE) sans qu'on les écrive. C'est le pattern DAO (Data Access Object) automatisé.

Q7 : Pourquoi @Transactional sur les méthodes service ?

R : @Transactional garantit que si une opération échoue en cours de route, toutes les modifications sont annulées (rollback). Par exemple, si on ajoute un événement et qu'on déduit le budget, les deux opérations doivent réussir ensemble ou échouer ensemble.

Q8 : Qu'est-ce que JSON et comment il est utilisé ?

R : JSON (JavaScript Object Notation) est un format texte pour échanger des données. Spring Boot sérialise automatiquement les objets Java en JSON (via Jackson) et les envoie via HTTP. Le client Swing les reçoit et les désérialise avec ObjectMapper.

Q9 : La base de données est où ?

R : La base PostgreSQL est hébergée sur Supabase (cloud). La connexion se fait via JDBC avec SSL (sslmode=require) sur le port 5432. Les identifiants sont dans application.properties. Hibernate gère la création et mise à jour des tables automatiquement (ddl-auto=update).

Q10 : Que fait DataInitializer.java ?

R : C'est un CommandLineRunner qui s'exécute au démarrage. Il vérifie si la base est vide, et si oui, insère des données de démonstration : 1 admin, 3 présidents, 3 clubs avec budgets, 30 étudiants, et quelques événements initiaux.

❓ Questions Avancées & Réponses

Q11 : Pourquoi ne pas utiliser Spring Security pour l'authentification ?

R : Pour un projet académique, Spring Security ajoute une complexité importante (tokens JWT, filtres HTTP). On a implémenté une authentification manuelle simple côté service : membre.authentifier(email, mdp). La méthode authentifier est un bon exemple d'encapsulation car elle compare les champs privés directement.

Q12 : Qu'est-ce que JPA et Hibernate ?

R : JPA (Jakarta Persistence API) est une spécification Java pour mapper des objets à des tables de base de données. Hibernate est l'implémentation concrète utilisée par Spring Boot. Les annotations @Entity, @Table, @OneToMany, etc. viennent de JPA. Hibernate génère et exécute le SQL automatiquement.

Q13 : Que se passe-t-il si deux utilisateurs se connectent en même temps ?

R : Spring Boot gère le multi-thread automatiquement. Chaque requête HTTP est traitée dans un thread séparé. L'annotation @Transactional garantit l'isolation des transactions grâce à PostgreSQL.

Q14 : Pourquoi @JsonIgnore sur le mot de passe ?

R : Pour des raisons de sécurité. Sans @JsonIgnore, Jackson inclurait le mot de passe dans toutes les réponses JSON. Avec cette annotation, le champ password est exclu de la sérialisation JSON, donc jamais envoyé au client.

Q15 : Qu'est-ce que cascade = CascadeType.ALL sur Budget ?

R : Cela signifie que toutes les opérations JPA (persist, merge, remove, refresh) sur le Club sont propagées automatiquement au Budget. Donc quand on sauvegarde un club, son budget est aussi sauvegardé. Quand on supprime un club, son budget est aussi supprimé. C'est la composition en pratique.

Q16 : Qu'est-ce qu'un @ManyToMany et comment est-il stocké ?

R : La relation @ManyToMany entre Club et Membre crée une table de jointure club_membres avec deux colonnes : club_id et membre_id. Un membre peut appartenir à plusieurs clubs et un club peut avoir plusieurs membres.

Q17 : Comment le discriminateur fonctionne avec SINGLE_TABLE ?

R : Hibernate ajoute une colonne role (pour Membre) ou type_evenement (pour Activite) dans la table. À la lecture, il regarde cette colonne pour instancier la bonne sous-classe Java (Admin, President, EvenementSportif, etc.). C'est le pattern discriminateur.

Q18 : Quelle est la différence entre @OneToMany et @ManyToOne ?

R : Ce sont les deux côtés de la même relation :
  • Club a @OneToMany vers Activite (un club → plusieurs événements)
  • Activite a @ManyToOne vers Club (plusieurs événements → un club, via club_id)
    • La colonne club_id existe dans la table activite, c'est le côté @ManyToOne qui possède la clé étrangère.

Q19 : Pourquoi avoir créé ExportUtils comme classe séparée ?

R : C'est le principe de Séparation des Responsabilités (Single Responsibility Principle). La logique d'export n'appartient pas à l'interface (AdminPanel), donc elle est extraite dans une classe utilitaire réutilisable par tous les panneaux.

Q20 : Que se passe-t-il si Supabase est inaccessible ?

R : L'application affiche une erreur de connexion au démarrage. Spring Boot ne peut pas démarrer sans connexion à la base. Dans un contexte de production, on ajouterait un mécanisme de reconnexion ou une base locale de secours (ex: H2 in-memory).
