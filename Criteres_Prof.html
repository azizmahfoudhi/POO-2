📌 Critère 1 — Application développée en Java ✅

Ce que la prof veut : Une application Java fonctionnelle. Ce qu'on a :
  • L'application entière est en Java 17
  • Interface graphique : Java Swing (fenêtres, boutons, tableaux)
  • Backend : Spring Boot (Java)
  • Point d'entrée : ClubApplication.java
    • Réponse si elle demande : "Toute l'application est en Java 17. L'interface graphique utilise Java Swing avec FlatLaf pour le design moderne, et le serveur backend est développé avec Spring Boot, également en Java."

    📌 Critère 2 — Connexion à une base de données ✅

    Ce que la prof veut : Persistance des données dans une base. Ce qu'on a :
  • Base de données : PostgreSQL hébergée sur Supabase (cloud)
  • Configuration dans application.properties :
    • ``` spring.datasource.url=jdbc:postgresql://aws-0-eu-west-1.pooler.supabase.com:5432/postgres spring.datasource.username=postgres.rvbcdnsclorkontygtza spring.jpa.hibernate.ddl-auto=update ```
  • 5 tables créées automatiquement par Hibernate : membre, club, budget, activite, club_membres
  • Connexion sécurisée via SSL (sslmode=require)
    • Réponse si elle demande : "On utilise PostgreSQL hébergé sur Supabase (cloud). La connexion est configurée dans application.properties. Hibernate génère et met à jour les tables automatiquement au démarrage grâce à ddl-auto=update."

    📌 Critère 3 — Couche backend avec Spring Boot ✅

    Ce que la prof veut : Un backend Spring Boot bien structuré. Ce qu'on a :
  • @SpringBootApplication dans ClubApplication.java
  • 4 contrôleurs REST : ClubController, MembreController, EvenementController, BudgetController
  • 3 services : ClubService, MembreService, EvenementService
  • 4 repositories : ClubRepository, MembreRepository, ActiviteRepository, BudgetRepository
  • Architecture en 3 couches : Controller → Service → Repository
    • Réponse si elle demande : "Le backend est structuré en 3 couches. Les Controllers exposent les endpoints REST HTTP. Les Services contiennent la logique métier. Les Repositories s'occupent de l'accès aux données via Spring Data JPA."

    📌 Critère 6 — Encapsulation ✅

    Ce que la prof veut : Attributs privés, accès via getters/setters. Ce qu'on a : 
    
// Budget.java — exemple parfait d'encapsulation
public class Budget {
    private double solde;          // ← attribut PRIVÉ

    public double getSolde() { return solde; }          // getter
    public void setSolde(double solde) { this.solde = solde; }  // setter

    // Méthodes métier encapsulant la logique
    public void ajouterDepense(double montant) { solde -= montant; }
    public void ajouterRevenu(double montant) { solde += montant; }
    public boolean aSuffisammentDeFonds(double montant) { return solde >= montant; }
}
    Autre exemple : Membre.java — le champ password est private avec @JsonIgnore pour empêcher sa sérialisation JSON. Réponse si elle demande : "Tous les attributs de nos classes sont privés. L'accès se fait uniquement via des getters et setters. Par exemple dans Budget, le solde est privé et on passe par des méthodes comme ajouterDepense() et ajouterRevenu() pour le modifier."

    📌 Critère 7 — Héritage ✅

    Ce que la prof veut : Utilisation de l'héritage entre classes. Ce qu'on a :

    Hiérarchie des Membres

    
Membre (classe mère)
├── Admin        (hérite de Membre)
├── President    (hérite de Membre)
└── Etudiant     (hérite de Membre)

    Hiérarchie des Événements

    
Activite (classe abstraite)
├── EvenementAcademique  (hérite de Activite)
├── EvenementSportif     (hérite de Activite)
├── EvenementCulturel    (hérite de Activite)
└── EvenementProposition (hérite de Activite)
    Stockage JPA : @Inheritance(strategy = InheritanceType.SINGLE_TABLE) — une seule table avec un discriminateur. Réponse si elle demande : "On a deux hiérarchies d'héritage. La première : Membre est la classe mère, Admin, President et Etudiant en héritent. La deuxième : Activite est une classe abstraite dont héritent EvenementAcademique, EvenementSportif et EvenementCulturel."

    📌 Critère 8 — Polymorphisme ✅

    Ce que la prof veut : Des méthodes redéfinies différemment selon les sous-classes. Ce qu'on a : 
    
// Activite.java — méthode ABSTRAITE (polymorphisme)
public abstract double getCoutTotal();

// EvenementAcademique → coût global fixe
@Override
public double getCoutTotal() { return coutGlobal; }

// EvenementSportif → équipement + logistique
@Override
public double getCoutTotal() { return coutEquipement + coutLogistique; }

// EvenementCulturel → décoration + sonorisation
@Override
public double getCoutTotal() { return coutDecoration + coutSonorisation; }
    Utilisation dans le service : 
    
// EvenementService.java — appel polymorphique
club.getBudget().ajouterDepense(evenement.getCoutTotal());
// Sans savoir si c'est académique, sportif ou culturel !
    Réponse si elle demande : "La méthode getCoutTotal() est abstraite dans Activite. Chaque sous-classe la redéfinit différemment : pour un événement académique c'est le coût global, pour un sportif c'est équipement + logistique, pour un culturel c'est décoration + sonorisation. Dans le service, on appelle getCoutTotal() sans connaître le type exact — c'est le polymorphisme."

    📌 Critère 9 — Architecture POO bien structurée ✅

    Ce que la prof veut : Code organisé selon les principes POO. Ce qu'on a :
  • Séparation des responsabilités : chaque classe a un rôle unique
  • Pattern MVC enrichi : Model / Controller / Service / Repository / View (Swing)
  • Classe abstraite Activite — impossible d'instancier directement
  • Constructeurs protégés sur les classes mères
  • Méthodes métier dans chaque entité (ex: Club.ajouterMembre(), Budget.ajouterDepense())
  • Relations JPA correctement modélisées : @OneToOne, @OneToMany, @ManyToMany
